## What Was Built

**Change A — promoteSubjects root-cause fix:**
The LLM prompt now shows a NUMBERED schema list (`0: Alpha Schema (12 members)`, `1: Beta Schema (8 members)`, ...) and asks for `relatedSchemaIndexes: [0, 3, 7]` (integer indices). The parse maps each index `k` to `schemas[k].schemaId`, filters out-of-range indices, deduplicates, and constrains to `schemaIdSet`. Backward-compat: if the LLM returns `relatedSchemaIds` (old UUID format), the existing `schemaIdSet.has(id)` filter still applies. A new `opts?: { force?: boolean }` parameter bypasses the Stage-1 gate check for force re-promotion. The Phase C transaction now also corrects node_scope for existing/refreshed subjects from stale slug-scope back to the project scope.

**Change B — subject→chapter containment:**
After the hub→subject containment block in `deriveDocGraph`, a new loop builds a `slugToDocId` map from all doc nodes, then for each subject's schemaIds checks if `UUID_RE.test(schemaId)` and `slugToDocId.has(schemaId)`. When both hold, a `doc_containment` edge srcId=subject.node.id → dstId=chapterDocId, w=1.0 is added (unless already in containmentSet). A chapter may receive edges from multiple subjects (DAG). These edges flow through the existing Phase B wipe-and-rebuild transaction.

**Change C — backfill-subjects CLI:**
New `src/adapter/backfill-subjects-cli.ts` following the same lock discipline as `promote-corpus-cli.ts` (validate DB path BEFORE acquireLock; release in finally). Builds a judge-tier provider via `resolveProviderOverlay(env, 'RECENSE_JUDGE_PROVIDER')` + `DefaultModelProvider({ generateConfig: judgeConfig, ... })` mirroring `run-sleep-pass.ts` L420+L457. Calls `promoteSubjects(scope, { force: true })` for each scope (default: `brain-memory,vtx,tonos`). Emits JSON to stdout. Registered in `recense.ts` dispatcher with `spawnScript('backfill-subjects-cli.js', process.argv.slice(3))` adjacent to `derive-doc-graph`, and added to the help string.
