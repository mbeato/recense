# Phase 39.2: Multi-Level Corpus Graph from Schema Projection - Context

**Gathered:** 2026-06-23
**Status:** Ready for planning

<domain>
## Phase Boundary

Turn the corpus doc layer from a shallow 2-level hub→subject **star** into a multi-level, interconnected doc→doc **graph**, derived purely as a **projection of the existing schema/fact graph** — LLM-free, ~$0, no re-ingest, no re-embed, runs on docs already in the live DB.

**Origin:** 39.1-05 live-verify found `doc_reference` near-empty (4 edges) — every doc relates only to one of two project hubs, while the rich structure already exists one layer down (2018 `abstracts` edges + shared facts/entities; subjects carry `subject-schema-ids` meta).

**In scope (from ROADMAP.md):**
1. Derive `doc_reference` edges subject↔subject + cross-project from shared schema-facts/entities + abstraction-ladder adjacency.
2. Multi-level + non-strict/many-to-many containment derived from the `abstracts` ladder.
3. The 2D corpus reader foregrounds the doc→doc graph.

**Out of scope — STRUCTURE ONLY (deferred to a later corpus-CONTENT phase):**
- Fixing empty/hollow subject docs (the `gatherFactsForSubject` bug).
- Exhaust-themed subject naming / the exhaust-rejection LLM-judge.
- Killing the ~200 regenerated schema-UUID chapter docs.
This phase adds doc→doc **edges and presentation**; it does not touch doc bodies, naming, or the promotion judge.

**Constraints (project invariants — hold throughout):** LLM-free, graph-as-truth (vector is derived cache), net-zero new runtime deps, online paths stay fast, never strengthen a fact from inferred output (D-43), clustering signal uses only gated members (D-37).

</domain>

<decisions>
## Implementation Decisions

### Reference Edge Derivation
- **D-01:** **Primary signal = discrete shared-members.** For each subject, expand `subject-schema-ids:<slug>` → schema members (gated facts/entities only, per D-37). Two subjects get a `doc_reference` edge weighted by how many member facts/entities they SHARE. Evidence-grounded and explainable; junk subjects with empty member sets naturally get zero edges. NOT centroid-cosine (rejected — cosine over noisy/exhaust centroids relates junk).
- **D-02:** **Abstraction-ladder adjacency is an additive second reference signal.** Two subjects whose schemas are connected by a `schema_rel` edge (cosine ≥0.72) or share a common super-schema parent also get a reference edge / weight boost — even when their member sets don't literally overlap. Catches conceptually-adjacent subjects (e.g. "retrieval"↔"ranking").
- **D-03:** **Density control = IDF down-weight + top-K per doc.** Weight each shared member by inverse frequency — a fact/entity shared across many subjects contributes ~nothing (kills the "`brain-memory` entity connects everything" hairball at the source); a rare co-occurring fact contributes a lot. Then keep only the top-K strongest reference edges per doc (like a wiki "related pages" box) for bounded out-degree.
- **D-04:** **Cross-project reference edges allowed, same rule, no special-casing.** Because IDF already suppresses ubiquitous generic members, only genuinely shared *substantive* facts/entities can bridge a `brain-memory` subject to a `vtx` subject. One code path; cross-project bridges are a feature, not a special case.
- **D-05:** **Reference symmetry = union** (default, research-flagged). Keep a reference edge if EITHER side ranks the other in its top-K (good "related" recall; top-K already bounds degree so union won't hairball). Planner may revisit mutual-only if union proves too dense on live data.

### Multi-Level Containment
- **D-06:** **Containment direction = abstraction-ladder position.** The parent is the doc whose schemas sit HIGHER in the `abstracts` ladder: if subject A's schemas are ancestors (via `abstracts` edges / super-schema membership) of subject B's schemas, A contains B. Mass-gap is used only as a fallback where the ladder is silent. ("Generality = real hierarchy," not raw evidence-mass.)
- **D-07:** **True multi-parent DAG (non-strict, many-to-many).** A subject contained by more than one broader doc gets multiple `doc_containment` edges — the corpus is a DAG, not a tree (e.g. "embeddings" under both "retrieval" and "storage"). Viz/reader must handle a node with multiple parents.
- **D-08:** **Hub indexes every subject directly.** Keep a hub→subject containment edge for ALL of a project's subjects regardless of subject→subject depth edges — the hub stays a complete flat index; subject→subject edges add depth on top. (A subject simply has both its hub AND its broader-subject as parents — consistent with the DAG model. Hub doc never goes incomplete.)
- **D-09:** **Subject→subject ancestry rule = strict ALL** (default, research-flagged). A contains B only when *every* one of B's schemas has an ancestor in A's schema set — sparse but trustworthy hierarchy, matching the project's precision-first pattern. Planner/researcher should validate against live data and may loosen to a majority threshold if strict-ALL yields too few containment edges. Acyclicity must be guaranteed (the `abstracts` ladder is a DAG by construction; the mass fallback must not introduce cycles).

### Junk-Doc Handling & Edge Ownership
- **D-10:** **Projection ranges over ALL live docs; the viz does the filtering.** Derive edges for every live doc including the ~200 UUID schema-chapter docs (keeps the signal complete). The UUID docs are left in the DB untouched — out of scope to delete — and when the later content phase kills them, wipe-and-rebuild drops their edges automatically. Empty-body subjects ARE included (linkage comes from `subject-schema-ids` meta, not prose).
- **D-11:** **New `DocGraphDeriver` owns ALL `doc_reference` + `doc_containment` edges.** It wipe-and-rebuilds them every sleep pass (mirrors `SchemaRelationDeriver`). Retire corpus-promoter's old centroid-cosine reference/containment creation and fold `promoteScope`'s hub→chapter containment into the new deriver's output. ONE authority for doc-graph structure — no double-derivation.
- **D-12:** **`doc_link` (prose citations) stays untouched.** It is a different EdgeKind from a different source (`doc-writer.ts`, citation verification) and is NOT owned by the new deriver.

### Viz Foregrounding (2D corpus reader)
- **D-13:** **Default state: everything shown, UUID chapter docs dimmed.** All docs render on open; the ~200 UUID schema-chapter docs are de-emphasized (faded/small) but visible — no hidden toggle state. Honest presentation; junk de-prioritized without concealment.
- **D-14:** **All three edge types visually distinct.** Containment = directed hierarchy edge (arrowed, heavier weight); reference = peer link (thin, undirected); cross-project reference = its own treatment (distinct color or dashed). Reads the multi-level structure AND cross-project bridges at a glance.
- **D-15:** **Layout = force-directed, hubs anchored.** Project hubs act as gravity anchors so each project's subjects pool around their hub and cross-project bridges stretch between regions. Reuse/retune the existing d3-force.
- **D-16:** **Node coloring grouped by project scope.** Each project hub and its subjects share one hue (derived from `node_scope`) — stable (a node's color changes only if its scope does), reinforces the hub-anchored pools, and makes cross-project reference edges visibly span color regions. Replaces today's all-nodes-one-color.
- **D-17:** **Scope colors via generative hue rotation, within a constrained muted band.** Programmatically rotate hue so any number of onboarded projects auto-gets a distinct tint (scales without hand-tuning). GUARDRAIL: bound saturation/lightness to the founder-locked muted rose/slate/mauve family and reserve warm amber for selection/activation only — auto-generated tints must stay on-brand. The "dimmed" chapter treatment (D-13) composes as a low-opacity/desaturated version of the node's scope tint; hub renders as the saturated anchor of its scope's tint.

### Surfacing & Rollout
- **D-18:** **Reader-panel surfacing = yes.** Feed the new projection edges into Phase 39's existing related/backlinks panel (`stmtDocBacklinks` already reads doc edges — near-free). The graph viz is not the only beneficiary; the prose reader's "related pages" gets richer too.
- **D-19:** **Live backfill CLI = yes.** Add a manual one-shot CLI (mirrors `promote-corpus-cli.ts` / `generate-doc-cli.ts`) so the projection can backfill + be inspected on the live brain immediately, without waiting hours for an organic sleep pass. The same deriver also runs automatically in the sleep pass (D-20).

### Claude's Discretion
- **Insertion point:** The `DocGraphDeriver` runs in `consolidate()` Phase C, immediately AFTER `corpusPromoter.promote()` (`consolidator.ts` ~L916) and before `insightReflector.reflect()` / eviction — by then doc nodes exist, `subject-schema-ids` meta is persisted, and schema/super-schema structure is fresh. (**D-20.**)
- **Rebuild discipline:** Single atomic transaction, wipe-and-rebuild from scratch each pass (no incremental), matching `SchemaRelationDeriver` — it's a derived cache, graph is truth.
- **IDF document-frequency scope** (corpus-wide vs per-project), exact **top-K value** (~6–8), `schema_rel`/super-schema adjacency weight vs shared-member weight balance, and edge-weight normalization — tuning details for research/planning.
- **Containment vs reference de-dup:** if a pair qualifies for both, containment should suppress the redundant reference edge between the same pair (sensible default; planner confirms).
- **Edge metadata:** whether to store the contributing shared-member count / IDF score as edge weight `w` for viz opacity (D-14/D-17 want weight-driven styling) — planner's call.

<specifics>
## Specific Ideas

- The whole phase is a **projection of the existing graph** — "the rich structure already exists one layer down; surface it." No new knowledge is created, only edges that re-express what the schema/fact graph already encodes.
- The reference graph should read like a **wiki "related pages" box** — bounded, relevant, explainable ("these docs share N facts"), not a similarity hairball.
- Color: "different neighborhoods having some sort of different color" — project scope = neighborhood, tints stay within the in-app palette (matches the standing brain note to align node colors with the app palette).
- Cross-project bridges are a deliberate feature: a substantive fact shared between `brain-memory` and `vtx` should visibly stretch a colored edge between the two project pools.
