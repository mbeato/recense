<objective>
Close GAP-02 (MEDIUM): an executed or rejected P0 re-proposes on every push tick (observed: fresh DeepSeek proposal cards at 15:42 / 15:44 / 15:46 after a 15:40 Approve+execute), throttled only by the daily cap. Root cause: the approval handler writes the `hitl` audit but never records a terminal `surfaced_event` outcome for the occurrence, so `surface()` keeps returning the item until its deadline passes or the cap is hit.

Fix: carry the `node_id` on the StoredProposal (the occurrence due_at is already stored as `dueAt`), and on a terminal decision call `surfaceSeen({outcome:'completed'|'dismissed'})` for `(node_id, occurrence_due_at)` so the occurrence is excluded from re-surfacing — making the daily cap a backstop, not the only brake (D-09 / ACT-01).

Purpose: Stop wasted DeepSeek calls and duplicate approval cards for an already-acted P0; align proposal terminal decisions with the Phase-22 surface-seen dedup the plain-notify path already uses.
Output: StoredProposal.nodeId + terminal surfaceSeen on execute-success and reject + client unit tests.

Scope discipline (CLAUDE.md): surgical, client-only (zero src/ imports — CLIENT-01). Snooze stays non-terminal (re-offer is deferred per CONTEXT.md). A FAILED execute (isError) is NOT marked completed — the proposal is already removed (at-most-once) and the item may legitimately re-surface for a retry, bounded by the daily cap; only a SUCCESSFUL execute is terminal.
</objective>
