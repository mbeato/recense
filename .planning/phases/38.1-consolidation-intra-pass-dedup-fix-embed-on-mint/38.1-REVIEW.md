# Phase 38.1: Code Review Report

**Reviewed:** 2026-06-21T15:20:00Z
**Depth:** standard

## Summary

Reviewed the embed-on-mint intra-pass dedup fix (DEDUP-01): a new optional `ClaimDecision.claimVec` field, two fill-site assignments (auto-unrelated slot + judge-escalated slot), 8 `setEmbedding(mintedId, decision.claimVec)` calls across the mint paths in `applyDecision`, and the new DEDUP-01 regression test.

The core fix is correct and the load-bearing invariants hold:

- **T-01-DIRTY (single writer):** Confirmed. No raw `UPDATE` on `node.embedding` was added — all 8 sites route through `store.setEmbedding`, which remains the only embedding writer. `setEmbedding` stamps `embedded_hash = value_hash`, marking the freshly-minted node clean so Phase C `reembedDirty()` does not redundantly re-embed it.
- **T-02-ASYNC (no await in transaction):** Confirmed. `applyDecision` is synchronous; the 8 added `setEmbedding` calls are synchronous primitives executing inside the Phase B `db.transaction(...).immediate()` block (lines 813-829). No `await` was introduced inside any transaction.
- **Mint-only targeting:** Confirmed. Every `setEmbedding(..., decision.claimVec)` call immediately follows a `newId()` + `upsertNode({ value: decision.claimValue })` mint. None targets a tombstoned or pre-existing node. Embedding text (`claim.value` → `claimVec`) is byte-identical to the minted node's `value` (`decision.claimValue`), so the stamped vector is consistent with what `reembedDirty` would have computed from `node.value`.
- **DEDUP-01 is a true regression guard:** Verified empirically. I ran the test against the base commit's `consolidator.ts` (without the fix) and it **fails** (`tombstoned.cnt === 0`, judge never escalated); against the fixed source it **passes**. The T-FK-01 null-coercion path does not mask it — the candidate-capturing judge echoes `candidates[0].id` (the runtime-minted node id, which is in the candidate set), so the guard passes and the contradict verdict reaches `applyDecision`. Without the fix, episode A's minted node has `embedding=NULL`, is invisible to episode B's topk, B auto-unrelates, and no judge call ever happens.

Typecheck passes. Findings below are robustness and maintainability concerns, not correctness defects in the shipped behavior.

## Warnings

### WR-01: DEDUP-01 test relies on `judge()` override that the production default path does not call directly — silent coverage gap if MockModelProvider.judgeBatch is refactored

**File:** `tests/consolidator.test.ts:435-451` (CandidateCapturingProvider), `src/model/provider.ts:228-236` (judgeBatch)

**Issue:** The consolidator never calls `provider.judge()` directly — the default (non-`RECENSE_ENABLE_JUDGE_BATCH`) path calls `provider.judgeBatch([single])` per pending claim (`consolidator.ts:741-747`). The test's `CandidateCapturingProvider` overrides `judge()`, and it only gets exercised because `MockModelProvider.judgeBatch` happens to delegate to `this.judge()` in a loop (`provider.ts:233`). This delegation is an undocumented implementation detail of the mock. If `judgeBatch` is ever refactored to a queue-based implementation (mirroring `judge`'s own `judgeQueue` path), the override becomes dead code, `judgeBatch` would hit the empty `judgeScript: []` queue and throw "judge queue exhausted", and the test would start failing for an unrelated reason — or worse, mask the regression it is meant to guard.

**Fix:** Override `judgeBatch` directly (or in addition to `judge`) in `CandidateCapturingProvider` so the test binds to the method the consolidator actually invokes:
```ts
override async judgeBatch(
  items: Array<{ claim: string; candidates: Array<{ id: string; value: string }> }>,
): Promise<JudgeVerdict[]> {
  return Promise.all(items.map(i => this.judge(i.claim, i.candidates)));
}
```
This makes the test independent of MockModelProvider's internal delegation.

### WR-02: DEDUP-01 test inherits `RECENSE_TYPED_EXTRACTION_MODE` from the prior suite's leaked env state (cross-suite env contamination)

**File:** `tests/consolidator.test.ts:455` (DEDUP-01 describe block), `tests/consolidator.test.ts:127-141` (prior suite's beforeEach/afterEach)

**Issue:** The first suite (`Consolidator: typed-edge extraction`) sets `process.env['RECENSE_TYPED_EXTRACTION_MODE'] = 'merged'` in `beforeEach` and restores it in `afterEach`. The new DEDUP-01 suite has no `beforeEach`/`afterEach` for this env var and does not assert its state. Vitest runs all describe blocks in the same process, and env restoration depends on `savedTypedMode` having captured the pre-suite value. If the developer's shell exports `RECENSE_TYPED_EXTRACTION_MODE`, or test ordering/parallelism changes, the DEDUP-01 episodes (source defaults to a typed-extraction-eligible source) could take the merged/separate extraction path instead of the intended bare-array path. The test currently passes only because the ambient value is unset and the afterEach correctly deletes it — but this is incidental, not defended. The `generateScript` provides bare-array JSON (`JSON.stringify([{...}])`), which `parseMergedExtraction` would not parse as `{facts, triples}` — so a leak would silently yield zero claims, B never escalates, and the test would fail confusingly rather than guard the regression.

**Fix:** Pin the env var explicitly in the DEDUP-01 suite so its extraction path is deterministic regardless of ambient state:
```ts
describe('DEDUP-01: embed-on-mint intra-pass visibility', () => {
  let saved: string | undefined;
  beforeEach(() => { saved = process.env['RECENSE_TYPED_EXTRACTION_MODE']; delete process.env['RECENSE_TYPED_EXTRACTION_MODE']; });
  afterEach(() => { if (saved === undefined) delete process.env['RECENSE_TYPED_EXTRACTION_MODE']; else process.env['RECENSE_TYPED_EXTRACTION_MODE'] = saved; });
  // ...
});
```

## Info

### IN-01: Eight near-identical `if (decision.claimVec) { this.store.setEmbedding(...) }` sites invite drift on the next mint-path addition

**File:** `src/consolidation/consolidator.ts:945, 973, 997, 1052, 1081, 1103, 1151, 1174`

**Issue:** The embed-on-mint guard is copy-pasted verbatim at 8 mint sites. This is the same duplicated-logic-across-sites pattern the file header repeatedly calls out as the root cause of prior bugs (FK-01/FK-02, and the `isEligibleForExtraction`/`markSkipped` centralizations created specifically to prevent it). A future ninth mint path (e.g. a new contradiction variant) can silently omit the `setEmbedding` call and reintroduce the exact NULL-embedding-invisibility bug this phase fixes, with no compile-time or test signal unless a new regression test happens to cover that path.

**Fix:** Extract a private helper that mints + stamps in one call, e.g. `mintNode(decision, { prev_value? })` returning the new id, that internally calls `upsertNode` then `if (decision.claimVec) setEmbedding(id, decision.claimVec)`. Every mint site becomes a single call, making it structurally impossible to mint without stamping. (Out of scope for a surgical fix; noting per the project's "duplicated logic becomes drift" convention.)

### IN-02: `claimVec` is absent on the fast-path confirm and judge-fill-via-missing-queryVec edge, but the contract comment over-promises "Undefined for fast-path confirms"

**File:** `src/consolidation/consolidator.ts:151-157` (field doc), `src/consolidation/consolidator.ts:778`

**Issue:** Minor doc/behavior nuance. The field comment says `claimVec` is "Undefined for fast-path confirms (no mint) and when claimVec is absent (guard)." The judge-fill site (line 778) uses `claimVecs[pendingJudges[i]!.slotIdx] ?? undefined`. `claimVecs[slotIdx]` can only be falsy if the embed call returned a short array — but the loop at line 610 already `continue`s when `!queryVec`, so any claim that reached `pendingJudges` necessarily had a defined `queryVec` at that index. The `?? undefined` is therefore dead-defensive (never falsy in practice). Harmless, but the guard implies a reachable null case that cannot occur given the upstream `if (!queryVec) continue`. No fix required; flagging so a future reader does not infer a real null-vec path through the judge branch.
