# Phase 39.1 Plan 02: Subject Promotion Exhaust-Gate Engine Summary

SubjectPromoter with two-stage exhaust-gate: LLM-free Stage-1 (mass + drift gates) and idempotent Stage-2 (one Sonnet call per scope when a gate is open), creating hub + subject stubs in one IMMEDIATE transaction with doc_containment edges and subject-schema-ids meta persistence.

## What Was Built

**Task 1 — Stage-1 LLM-free gates + SubjectPromoter scaffold (commit `45cea48`):**

Added `SubjectPromoter` class to `corpus-promoter.ts` with five prepared statements compiled in the constructor (T-01-SQL), all guarded by the D-37 firewall (`origin != 'inferred'`):
- `stmtGetSchemaMembersInScope` — member count per schema+scope for CREATE gate
- `stmtDriftCount` — count of `abstracts` members with `last_access > generated_at` for REFRESH gate
- `stmtGetLiveDocBySlug` — look up existing doc by `node_doc.slug`
- `stmtGetExistingSubjectSlugs` — existing `scope:*` slugs for the scope
- `stmtGetSchemasInScopeWithMass` — top-20 schemas by member count for Stage-2 input

Added `normalizeSubjectName(name)` (exported) — slug normalization with `[^a-z0-9]+→'-'` + trim.

`evaluateStage1Gates` separates CREATE (mass >= 4, no existing slug) from REFRESH (drift >= `corpusSubjectDriftThreshold`, independent of mass floor) — both SQL-only, zero LLM calls.

Created `tests/exhaust-gate.test.ts` (13 tests): Stage-1 LLM-free path (0 provider.generate calls asserted), REFRESH threshold at boundary (>= 3 fires; < 3 does not; exactly 3 fires), D-43 firewall (inferred members excluded from mass and drift counts), FTS suppression on new stubs.

**Task 2 — Stage-2 idempotent subject-proposal LLM call (commit `abfdf34`):**

The Stage-2 call in `promoteSubjects`:
- Builds prompt from top-20 schemas as `label: memberCount` lines.
- Includes `EXISTING SUBJECTS (do NOT rename these): ...` anchor (Pitfall 1 guard).
- Calls `provider.generate` once per scope, only when CREATE or REFRESH gate is open.
- Parses JSON `[{name, relatedSchemaIds}]` response with empty-output guard.
- Length-bounds each name to 200 chars (Security V5) before normalization.
- Matches normalized slug against existing slugs: existing → `refreshQueued`, new → `CREATE`.
- Filters `relatedSchemaIds` to only in-scope schema IDs (prevents LLM injection).

Created `tests/corpus-subject-promoter.test.ts` (12 tests): idempotency (existing slug → no new stub), new name → `scope:name` stub (never UUID slug), proposal not issued when no gate open, length-bound rejection (>200 rejected, 200 accepted), D-03 no-UUID-slug assertion, BLOCKER-2 meta key persistence for both created and refreshed subjects, stub invariants (value='', origin='inferred').

**Task 3 — Hub/subject stub creation + doc_containment + meta persistence + orchestration (commit `d275a29`):**

Phase C (single IMMEDIATE transaction, no `await` inside):
1. Creates hub stub (slug = scope, value='', origin='inferred', FTS-suppressed).
2. Creates each new subject stub (slug = `scope:name`, `node_scope.scope = project_scope` NOT schemaId — D-03 demotion).
3. Persists `subject-schema-ids:<subjectSlug>` meta via `store.setMeta(JSON.stringify(relatedSchemaIds))` for ALL accepted subjects (new + refreshed) — BLOCKER-2 contract for Plan 03.
4. Writes hub→subject `doc_containment` edges (fill-in-place, BUG-2c safe).

Fixed: `hubDocId` initialized to `''` (TypeScript strict-flow); `JudgeVerdict` mock updated with `relation`/`magnitude`/`judgeBatch` to satisfy `ModelProvider` interface.

Created `tests/corpus-hub-promoter.test.ts` (11 tests): hub slug = scope, doc_containment edges one-per-subject, edges between `type='doc'` nodes only, hub value='' + FTS suppressed, idempotency (second call creates no duplicate stubs), meta key format byte-identical (`subject-schema-ids:<slug>`), stub invariants (value='', origin='inferred', no node_fts), scope annotation = project scope NOT schemaId (D-03), no UUID slugs, inferred-only members don't open the gate (D-43).

## Deviations from Plan

**1. [Rule 1 - Bug] TypeScript strict-flow: hubDocId assigned inside transaction callback**

- **Found during:** Task 3 (tsc --noEmit)
- **Issue:** TypeScript infers `hubDocId` might not be assigned before `return { hubDocId }` since the assignment happens inside the transaction callback (which TypeScript can't prove always runs).
- **Fix:** Initialized `hubDocId` to `''` before the transaction. The transaction always assigns it; the empty string is a safe sentinel.
- **Files modified:** src/consolidation/corpus-promoter.ts
- **Commit:** d275a29

**2. [Rule 1 - Bug] JudgeVerdict mock missing required fields + ModelProvider missing judgeBatch**

- **Found during:** Task 3 (tsc --noEmit)
- **Issue:** The `JudgeVerdict` interface requires `relation` and `magnitude` fields; the `ModelProvider` interface requires `judgeBatch`. The mock providers in all three test files were missing these.
- **Fix:** Added `relation: 'unrelated' as const, magnitude: 0` to the judge return and `judgeBatch` stub method to all three test files.
- **Files modified:** tests/exhaust-gate.test.ts, tests/corpus-subject-promoter.test.ts, tests/corpus-hub-promoter.test.ts
- **Commit:** d275a29
