# Phase 39.1: Corpus Quality — project-hub & subject docs via zero-intervention LLM exhaust-gate, retroactive junk-doc cleanup, recense + vtx ingestion - Context

**Gathered:** 2026-06-22
**Status:** Ready for planning

<domain>
## Phase Boundary

Make the brain's *generated doc corpus* actually good. Four threads, sequenced:

1. **Restructure the doc taxonomy** — replace the landing-doc + schema-UUID-chapter-doc model with **project-hub docs** (synthesized overview + index) and **LLM-named subject docs**; schema clustering becomes an internal signal, not the doc unit.
2. **Zero-intervention LLM exhaust-gate** — a two-stage mechanism that harvests cheap signal from the sleep pass's existing extract/judge calls to decide what's worth a doc, then spends a dedicated generation call only when a create/refresh gate opens — with no user action.
3. **Retroactive junk-doc cleanup** — one-time removal of obsolete/structural junk docs already in the live brain.
4. **recense + vtx ingestion** — run the v6.0 `ingest-project` pipeline (full code survey) on recense-itself and the vtx project; this is the end-to-end validation that the new machinery works.

This phase clarifies HOW to implement this set. New capabilities beyond it belong in other phases. The goal in ROADMAP.md is currently "[Urgent work - to be planned]" — this CONTEXT.md establishes it; ROADMAP/REQUIREMENTS should be updated at plan time.

</domain>

<decisions>
## Implementation Decisions

### Doc Taxonomy (full restructure)
- **D-01:** Full restructure — abandon the landing-doc + schema-UUID-chapter-doc model as the doc unit. The doc unit becomes **project-hub** (one per project) + **subject docs** (many per project).
- **D-02:** **Subject docs are LLM-named.** During the sleep pass the LLM looks at the project's atoms/schemas/episodes and proposes a small set of named subjects (e.g. "retrieval", "sleep pass", "config"). Subjects emerge from content — they are NOT 1:1 with schema cluster IDs.
- **D-03:** **Schema clustering is demoted to an internal signal** (it informs mass/grouping) but is no longer the unit a doc maps to. This changes `doc-generator.ts` and `corpus-promoter.ts`.
- **D-04:** **project-hub doc = synthesized overview + index.** The LLM writes a real overview (what the project is, key facts, current state) AND the doc carries a linked index of its subject docs. Reader gets orientation + navigation in one. Richer than today's per-scope landing doc.

### Exhaust-Gate (zero-intervention generation)
- **D-05:** **Two-stage cost model.** Stage 1 = harvest cheap signal from the extract/judge LLM calls *already running* in the sleep pass (atoms, schema names, summaries) to DECIDE what's worth a doc — ~no extra LLM cost. Stage 2 = spend a *dedicated* generation call only when the gate opens. ("Both: cheap signal, gen on threshold" — most controllable cost.)
- **D-06:** **Two gates: create on mass, refresh on drift.** CREATE when a subject crosses a mass/member threshold (it's substantial). REFRESH when enough new/changed atoms accumulate since the doc was last generated (it has drifted from reality). This applies recense's prediction-error-gated update philosophy to docs — keeps the corpus fresh without churning every pass, and directly serves the core value ("stays correct over time").
- **D-07:** **Per-pass budget cap + priority queue.** Cap docs generated per sleep pass (extend the existing `RECENSE_CORPUS_GEN_MAX` = 25 knob). When more subjects qualify than the cap allows, prioritize by mass/staleness and defer the rest to the next pass — a self-draining queue. This guards against a surprise token bill on a big drift event or a bulk ingestion (founder cost-discipline: subscription tokens are real).

### Junk-Doc Cleanup (retroactive, one-time)
- **D-08:** **Junk = obsolete taxonomy + structural junk, deterministic.** Remove: (a) old schema-UUID chapter docs superseded by the restructure, (b) empty/never-filled stubs, (c) docs from noise-filtered schemas (commit/path/tmp patterns). All known-bad by structure — no LLM quality judgment, no false-positive deletion risk.
- **D-09:** **Execution: dry-run → approve → snapshot → hard-delete.** Emit a report of exactly what would be removed; founder approves; take a `VACUUM INTO` snapshot (reuse the nightly-hygiene snapshot path `~/.config/recense/snapshots/`) before deleting; then hard-delete. Safest for a one-time op on the live ~7000-node brain (the no-DB-backup SPOF). Hard-delete is acceptable because docs are `origin='inferred'`, regenerable nodes — NOT evidence-backed facts, so the "never delete evidence via decay" guard does not apply.

### Ingestion (recense + vtx)
- **D-10:** **Full code survey** — run `ingest-project`'s LLM repo survey on both recense-itself and vtx (the pipeline default the v6.0 phases 29–32 built). Dogfoods the pipeline and stress-tests the new exhaust-gate + noise filter on real, potentially-noisy survey output.
- **D-11:** **Sequencing: Build → clean → ingest.** (1) Restructure taxonomy + exhaust-gate, (2) retroactive junk cleanup of old docs, (3) ingest recense+vtx fresh so they generate docs directly in the new taxonomy and serve as end-to-end validation. Cleanest causality — ingesting before the new machinery exists would just produce old-taxonomy docs that immediately become junk; cleanup's "obsolete chapter doc" definition depends on the restructure being defined first.

### Claude's Discretion
- **Scope slugs** for ingestion: default recense → `brain-memory` (its cwd-derived scope), vtx → `vtx`. Planner may confirm/adjust. (Note the self-referential wrinkle: recense ingested into the recense brain is customer-zero dogfooding.)
- **Drift measurement mechanics** (how "enough new/changed atoms since last gen" is computed) — implementation detail for research/planning, not user-decided.
- **Hub↔subject link surfacing** in the reader — can reuse the Phase-39 index/backlinks surfaces rather than duplicating link lists; planner's call.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- **`generateCorpusDocs()` idempotency** — already fills-only-empty-stubs; the gated generation can layer create/refresh decisions on top without a rewrite of the write path.
- **`RECENSE_CORPUS_GEN_MAX`** — existing per-pass cap knob; D-07's budget cap extends it rather than inventing a new mechanism.
- **Nightly-hygiene snapshot path** (`VACUUM INTO` → `~/.config/recense/snapshots/`, keep 5) — D-09 cleanup reuses it verbatim as the pre-delete rollback point.
- **`ingest-project` pipeline** (survey → scope-tagged episodes → marker → sleep-pass auto-corpus) — D-10/D-11 ingestion uses it as-is.
- **Pending-corpus markers** — crash-safe marker pattern (retry next pass on failure) is the model for the exhaust-gate's deferred-overflow queue (D-07).

### Established Patterns
- **Phase-boundary / sleep-pass-deferred generation** — all LLM/doc cost lives offline in the sleep pass; online paths stay LLM-free. The exhaust-gate MUST stay inside this boundary.
- **PE-gated update (don't-rewrite)** — D-06's refresh-on-drift is the doc-level expression of the engine's core reconsolidation philosophy. Keep it consistent with that mechanism, not a bespoke staleness timer.
- **Docs are `origin='inferred'`, `s=0`, `c=1.0`, `embedding=NULL`, FTS-suppressed** — lifecycle-exempt, regenerable. This is *why* hard-delete cleanup is safe (D-09).
- **Noise filter (corpus-promoter)** — the existing junk-suppression mechanism; D-08 cleanup and D-10 ingestion both lean on it.

### Integration Points
- Exhaust-gate signal harvest taps the extract/judge results inside the consolidator before they're discarded (the "exhaust").
- Gated generation replaces/wraps the eager `generateCorpusDocs()` call in `run-sleep-pass.ts`.
- Cleanup is a new one-shot CLI (dry-run + snapshot + delete) operating over `type='doc'` nodes + `node_doc` sidecar.
- Ingestion is existing CLI invocations; the only new dependency is that it runs AFTER build + clean (D-11).

</code_context>

<specifics>
## Specific Ideas

- The phase name's "LLM exhaust" = the **waste heat of consolidation, recaptured** — reuse the extract/judge work the sleep pass already does to drive doc decisions, rather than paying for separate detection passes.
- "Zero-intervention" = the corpus stays good and fresh with NO manual `generate-doc` / `ingest-project --consolidate` runs — create/refresh gates fire automatically in the sleep pass.
- recense-into-recense ingestion is intentional customer-zero dogfooding (the founder's own brain documents its own engine).

</specifics>
