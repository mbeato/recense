### BLOCKER — ROOT-CAUSED + FIXED this session (debug `bulk-consolidation-stall`, fix `67b3ade`)

The stall was NOT a hang — it was an infinite stale-lock loop. `consolidator.ts` prefetched extractions for ALL eligible episodes in one blocking `await` before the per-episode commit loop started. With 308+ episodes at 4-concurrent × ~30s, the prefetch took ~38 min, exceeding `LOCK_STALE_MS` (30 min). The hourly launchd agent reclaimed the "stale" lock and relaunched from scratch with the same backlog — `markConsolidated` never ran. Small backlogs (≤18 eps ≈ 2.5 min) fit the window, which is why it only broke after the bulk ingest.

**Fix (FIX-STALL-01, committed `67b3ade`, `src/consolidation/consolidator.ts`):** chunk the prefetch into batches of 20 (env `RECENSE_PREFETCH_EPISODE_BATCH_SIZE`) — prefetch 20 → commit 20 → next batch. Progress is durable every ~2.5 min so a stale-lock restart resumes from a smaller backlog. Episode-ordering invariant preserved. tsc clean, 2147/2148 tests pass, confirmed LIVE (consolidated 2608→2609+ at 01:53Z). Debug session resolved at `.planning/debug/resolved/bulk-consolidation-stall.md`.

**Known follow-up (deferred):** the full ~313-episode drain is ~1.5–2 hrs (per-episode JUDGE call ~20-30s is the real bottleneck, not extraction). A single pass exceeds the 30-min lock-stale window, so a long pass needs the lock kept fresh. `heartbeatLock()` already EXISTS in `src/adapter/lockfile.ts:237` but is NOT called by the consolidator (only the watcher). Proper fix: thread a heartbeat callback through `consolidate()` (CLI owns the lock; src/consolidation has zero adapter imports — keep it that way via a callback param).
