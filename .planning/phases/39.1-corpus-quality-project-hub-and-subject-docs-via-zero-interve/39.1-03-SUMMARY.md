# Phase 39.1 Plan 03: Sleep-Pass Wiring + Hub/Subject Generation Loop Summary

Extended the generation loop to dispatch hub and subject docs by slug shape, enforced the D-07 budget cap with self-draining deferred markers, and wired promoteSubjects best-effort into the CORPUS-06 sleep-pass block.

## What Was Built

**Task 1 — Hub+subject dispatch in corpus-generator.ts + budget-cap test (commits `e425862`):**

Extended `generateCorpusDocs` to classify and dispatch empty stubs by slug shape:
- UUID slug → schema-chapter path (`generateDocForSchema`, D-03 backward compat; no longer the primary path)
- Bare scope string (no ':') → hub doc path (`generateDocForHub` with linked `{name, docId}[]` index built from `doc_containment` children, D-04 / BLOCKER-1)
- `scope:name` slug → subject doc path (`generateDocForSubject` with schemaIds rebuilt from `subject-schema-ids:<slug>` meta, BLOCKER-2 / D-05 constraint)

Priority sort before slicing: hubs (0) → subjects (1) → schema-chapters (2). Overflow stubs beyond `maxDocs` get `pending-subject-doc-gen:<slug>` meta markers (`store.setMeta`); markers cleared on success (`store.deleteMeta`) — crash-safe self-draining queue (D-07).

Also fixed `stmtFindLiveDocForSlug` in `doc-writer.ts` (Rule 1 bug): changed lookup from `ns.scope = ?` to `nd.slug = ?` so the BUG-2c fill-in-place logic correctly finds hub and subject stubs. The old `ns.scope` query broke for subject docs (where `slug = scope:name` but `ns.scope = project_scope`) and for hub docs when non-empty subject docs shared the same `ns.scope`.

Created `tests/corpus-budget-cap.test.ts` with 6 tests covering:
- Hub dispatch: linked `{name,docId}[]` index from `doc_containment` children (BLOCKER-1)
- Subject dispatch: meta-rebuilt schemaIds with zero extra LLM call (BLOCKER-2)
- Budget cap (3 stubs / maxDocs=1): 1 generated + 2 deferred markers written
- Self-draining: second pass generates next, clears one marker
- Hub vs subject prompt dispatch separation
- Missing meta key: log + skip, no LLM call

**Task 2 — promoteSubjects wiring in run-sleep-pass.ts (commit `cab85a0`):**

Added `SubjectPromoter` import and instantiation (with `RECENSE_CORPUS_SUBJECT_DRIFT_THRESHOLD` env override).

## Deviations from Plan

**1. [Rule 1 - Bug] doc-writer.ts stmtFindLiveDocForSlug query broken for new taxonomy**

- **Found during:** Task 1 (test corpus-budget-cap hub test failure — hub stub not filled)
- **Issue:** `stmtFindLiveDocForSlug` used `ns.scope = ?` to find existing stubs. For subject docs, `slug = scope:name` but `ns.scope = project_scope` — query returns nothing, so the fill-in-place path never fires (creates a new node instead, leaving the original stub empty). For hub docs, the query also returns non-empty subject docs that share `ns.scope = project_scope`, causing writeDoc to tombstone a subject stub instead of finding the empty hub stub.
- **Fix:** Changed `JOIN node_scope ns ON ns.scope = ?` to `JOIN node_doc nd ON nd.slug = ?` so the lookup is precise by slug for all doc types.
- **Files modified:** src/consolidation/doc-writer.ts
- **Commit:** e425862

**2. [Rule 2 - Missing critical functionality] Deferred-queue drain in run-sleep-pass.ts**

- **Found during:** Task 2 (plan specifies drain of pending-subject-doc-gen markers)
- **Issue:** The plan described adding a "drain scan" for deferred markers in the sleep pass. In practice, the drain is automatic: the deferred stubs are still empty and `generateCorpusDocs` picks them up via the empty-stub query; markers are cleared on success by `store.deleteMeta`. No separate scan-and-consume loop is needed.
- **Fix:** Added observability-only drain (log the pending count before generation) so operators can see the queue size. The actual drain happens implicitly through the generation loop's marker-clear on success.
- **Files modified:** src/consolidation/run-sleep-pass.ts
- **Commit:** cab85a0
